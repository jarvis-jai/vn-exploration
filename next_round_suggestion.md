
## 先盤點現況（以 repo 最終快照來看）

從 zip 來看，repo 目前主要是：

* 根目錄 `README.md`（定位清楚：地圖、非決策、繁中為主）
* 10 個分類資料夾，各自一個 `README.md`：

  * `genres/`, `narrative-structures/`, `player-experience/`, `failure-modes/`, `representative-works/`, `communities/`, `tooling/`, `monetization/`, `localization/`, `adjacent-genres/`

整體 **結構是好的**，而且內容也不只是空骨架：每個 README 都已經有「分類項目」與一些「條目描述」。

但也有一個很關鍵的現象：

> 目前幾乎都是「總表型 README」在變厚，還沒有大量長出「葉子頁」（例如 `genres/school-romance.md`、`tools/renpy.md` 這種獨立頁），也幾乎沒有外部連結/來源索引。

這會直接影響「跑十幾個小時」的邊際效益。

## 你猜得很準：這種任務通常 5–6 小時就到極限（除非改任務型態）

### 為什麼十幾個小時不一定更好？

因為在你現在這個 prompt 設計下，AI 很容易進入：

* **把分類清單補完整**
* **把每個條目寫成通用描述**
* **維持中立、不下結論**
* 然後就開始「換句話說」「微調格式」「補同義詞」

這種「地圖式內容」如果**缺少外部資料（links / citations / 例子）作為燃料**，會很快到達飽和點。
所以：**時間拉長 → commit 變多，不等於信息密度等比例上升。**

### 什麼情況下十幾小時才會顯著更好？

如果你把任務改成這種「可無限挖礦」型：

* 每輪都要新增 **5–15 個可點的連結**（作品頁、討論串、postmortem、工具文檔）
* 每輪都要新增 **1–2 個葉子頁**（不是只改總表 README）
* 每輪都要更新 `sources/visited.md`（避免重複逛同一圈）

那十幾個小時才會真的「越挖越多」。

## 驗收成效：我建議你用 3 個指標判斷「好不好」

### A. 密度指標：可點的外部連結密度

* 如果一晚過後，每個主題還是「概念描述」為主，連結很少
  → 代表大多時間在「生成通用知識」，價值開始遞減
* 如果連結很多、而且來源多樣
  → 十幾小時就很值

### B. 結構指標：葉子頁數量

* 只有總表 README 變厚
  → repo 會越來越像一篇長文章，反而不好用
* 有大量葉子頁（每個主題獨立一頁）
  → repo 會越來越像百科全書（你想要的）

### C. 可信度指標：是否混入「看似精確但未證實」的數字/結論

我在 `localization/README.md` 看到一些像「短篇/中篇字數範圍」這種**數字型內容**——這類很容易變成「AI 自己估的」。
這不是不能寫，但應該要：

* 標註為「推測/經驗值」或
* 後面加「需要來源」或
* 乾脆改成「待查證」

不然 repo 會慢慢混入「很像真的」的內容，長期會害你。

## 下一步怎麼補救/升級（讓下一個 8 小時真的變更好）

我建議你把今晚這種「骨架填充」改成「索引挖礦」：

### 方案 1：改 prompt 的「完成條件」＝每輪必須新增來源與葉子頁

把 cron 規則加兩條就會差很多：

* **每輪至少新增 8 個外部連結**（分散到不同 section）
* **每輪至少新增 1 個葉子頁**（新檔案，不只改 README）

並新增兩個 repo 固定檔案：

* `sources/visited.md`：記錄已看過的網域/頁面（避免反覆）
* `reports/run-log.md`：每輪寫 5 行：做了什麼、加了哪些頁、加了哪些 links、commit hash

### 方案 2：降低 commit 噪音（讓你明天比較好驗收）

75 commits 的問題不是「多」，是「你很難讀」。
你可以把 cron 改成：

* 每 8 分鐘跑一次沒問題
* 但要求它：**累積 3 次 run 才 commit 一次**（或至少一次 commit 要包含「一個主題的完整增量」）
